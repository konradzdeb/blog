Challenge

File manipulation is a frequent task unavoidable in almost every IT business process. Traditionally, file manipulation challenges are considered within the ramifications of specific tools native to a given system. As such, the one may consider writing and scheduling shell script to undertake frequent file operations or using more specific purpose-built tools like logrotate in order to archive and clean logs or tools like Kafka are used to build streaming-data pipelines. R is usually though of as a statistical programming or language or, more incorrectly, as an environment for a statistical analysis. R is a mature programming language. What constitutes a programming language is a valid question. Wikipedia offers somehow wide definition:

A programming language is a formal language comprising a set of instructions that produce various kinds of output. Programming languages are used in computer programming to implement algorithms.

Pros and Cons of Using R as an ETL solution

R wasn’t designed to replace ETL1 processes; nevertheless, as every other well-developed programming language R offers a set of robust tools for accomplising file manipulation, analysis and creation. In particular, R package ecosystem offers a layer that provides a clean interface for accomplishing traditional

Pros

  • The article assumes that consideration on whether to utilise R within a team for pure ETL tasks is usually given in the context of R language being utilised to greater or lesser degree to facilitate data modelling, analytical or statistical work. In that context utilising R yields benefits mostly associated with ability to utilise the existing knowledge. Furthermore, as discussed later, R development team will be able to leverage the existing knowledge or expertise in accomplishing tasks that would traditionally
  • R is an exceptionally flexible language and offered data structures lend themselves well for file manipulation
  • Ability for our production process to be migrated across the systems and configurations increases. Let’s consider a simple example of generating temporary files. Utilising R base function tempfile or equiavlent call from the fs package, fs::file_temp() allows for swift creation of temporary files. Achieving that on MS Windows using PowerShell could be done using specific cmdlet called New-TemporaryFile $tmp = New-TemporaryFile. When working in command prompt we would leverage %TEMP% and %RANDOM% environment variables to come up with something like set "uniqueFileName=%tmp%\bat~%RANDOM%.tmp", as discussed at lengths in the related Stack Overflow answer. The point is that exercises like that incur addittional maintaince cost. Whereas tempfile is straightfoward, well documented and understood by every R user, call:
@echo off
setlocal EnableExtensions

rem get unique file name 
:uniqLoop
set "uniqueFileName=%tmp%\bat~%RANDOM%.tmp"
if exist "%uniqueFileName%" goto :uniqLoop

won’t be obvious to maintainers not familiar with command prompt, even if the actual goal behind the two is identical from a production perspective (to provide process with a temporary path)

  • Thanks to the Rocker Project R plays exceptionally well with Docker which is conducive to reducing future maintainability costs and enhancing portability. Decent article covering technicalities of using Docker with R was provided by Colin Fay.2

  • R has rich ecosystem offering API connectivity. Thanks to Plumber R users can, with relatively little effort, generate own APIs. Packages like wbstats and Eurostat provide convenient wrappers for sourcing data from publicly available repositories (article World Bank and Eurostat). Those can be immensely time server if we consider building processes that rely on background external demographic/macroeconomic data being incorporated in our processes on regular basis. The subject of R / API integration merits a separate article.

  • File format converstion. Through packages like haven R offers rich interface to access data in different formats (Stata, SPSS, etc.). When working with bloated legacy processes that may required handling input originating from heterogeneous sources (think SAS binary, in-house Oracle database and some periodical figures produced by World Bank) the R flexibility may prove actual life saver. Still we should be careful not to use R as an excuse not to refactor old processes.

Cons

  • Using R for file manipulation is not suitable for processes that are mostly in vast majority ETL tasks. If our intention is to monitor a location for change and then conditionally copy files building R-based process will incur an unnecessary dependency, whereas more tools would

  • Achieving fine-grained control over common operations may be more difficult in R. Let’s consider rsync. The Internet is full of examples how to achieve properly defined goals using rsync, it is also possible to find Python implementation of the algorithm3. At the time of writing this article, the attempts to bring rsync functionality relied on command line tool.4

  • Multithreading, the thinking around multithreading and asynchronous computation in R oscilliates around finding more efficient ways to accomplish computational effort. Whereas packages like r-lib/async or r-lib/processx offer aproaches to run processes in the backrgound or perform asynchronous I/O operations; those solutions are applicable to more complex process. Achieveing parallel rsync execution can be easily achieved using xargs, as found on Stack Overflow5:

ls /srv/mail | xargs -n1 -P4 -I% rsync -Pa % myserver.com:/srv/mail/

Conclusion

The R-based file manipulation process are only efficient, from a production perspective, when applied to processes containing a substantial analytical/statistical component.
